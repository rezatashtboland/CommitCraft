# AI Guidelines — CommitCraft

> This file is the official source of truth for the project's development rules.
> **Before running any command, read this file in full and act accordingly.**

Current app version: **1.0.0**

---

## 1. General Architecture Principles
- The project is a **modular**, **cross-platform** Python CLI tool.
- Code must follow **SOLID** principles:
  - Each class/module has a single, clear responsibility (SRP).
  - Open for extension, closed for modification (OCP).
  - Dependencies are injected via Dependency Injection, not hardcoded.
- Layers must be separated: `UI` / `Service` / `Config` / `Git` / `AI`.
- Use abstractions and interfaces for swappable parts (e.g., the AI provider).

## 2. Cross-Platform Compatibility
- The Python launcher must work in both environments: use `python3` on Linux/WSL and `python` on Windows (auto-detect the executable).
- Read all Git and subprocess output with `encoding="utf-8"` and `errors="replace"` so that Persian/non-UTF-8 characters don't cause decode errors or thread crashes.
- Handle paths with OS-independent tools (`pathlib`), not manual strings.

## 3. Language & User Interface (i18n)
- The UI is **single-language**: either fully English or fully Persian.
- Bilingual labels (e.g., `"Remove / حذف"`) are **prohibited**.
- The default language on first run is **English**.
- Persian-specific dependencies (RTL, font rendering) are installed **only on demand, when Persian is first selected**.
- UI strings must not be hardcoded; keep them in a translation/string-source layer.

## 4. Configuration Management
- Settings are stored in a **persistent** config file and preserved across runs.
- Sensitive and changeable values (API token, URLs, retry logic, language) are managed through the settings menu, not hardcoded.
- The **Working Copy** path is stored separately, outside the main config file, but remains persistent across runs.
- The Working Copy path is requested after checking the config and validated as a real Git repository.

## 5. Git Logic
- The staging flow must handle **deletions, additions, and modifications** seamlessly.
- Never stage a file directly if it no longer exists on disk (to avoid the `pathspec did not match` error when committing deleted files).
- Commit messages are generated by the AI (gapgpt).

## 6. Code Commenting
- **All code must be commented.**
- Each function/class needs a short, clear docstring describing purpose, inputs, and outputs.
- Complex or non-obvious logic gets inline comments (the why, not just the what).
- Comment language must be consistent across the whole project.
- File names, functions, variables, and technical terms stay in English.

## 7. Versioning (Semantic Versioning)
The version follows `MAJOR.MINOR.PATCH` and increments **based on the size and complexity of changes**:

- **PATCH** (`x.x.+1`): bug fixes, small low-complexity changes, no change to public behavior.
- **MINOR** (`x.+1.0`): new backward-compatible features, or medium changes with notable complexity.
- **MAJOR** (`+1.0.0`): broad changes, breaking changes, or architectural redesign.

Rules:
- Keep the version in a central constant (e.g., `__version__`).
- Before each significant commit, bump the version to match the scope of change.
- Reference the version change in the commit message.

## 8. Quality & Safety
- Handle errors transparently, never with a silent crash.
- Validate user input.
- Never print sensitive values (tokens) in output/logs.

## 9. Documentation Sync Rule
- Whenever a new feature, command, option, or setting is added to the project, or an existing behavior is changed, README.md MUST be updated within the same change	to reflect that feature. A code change that introduces user-facing functionality
- without a corresponding README.md update is considered incomplete.
- The README update must include the purpose of the feature, how to use it, and an example where appropriate. It must stay consistent with the existing language, 	tone, and structure of README.md, and must not document anything that does not exist in the code.
